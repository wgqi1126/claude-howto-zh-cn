# 代码重构 Skill

基于 Martin Fowler《重构：改善既有代码的设计》（*Refactoring: Improving the Design of Existing Code*，第 2 版）的系统化重构方法。本 Skill 强调在测试保护下进行安全、增量式改动。


> 「重构是在不改变代码外在行为的前提下，改变软件系统内部结构的过程。」—— Martin Fowler

## 核心原则

1. **行为不变**：对外行为必须保持不变  
2. **小步前进**：每次只做微小、可验证的改动  
3. **测试驱动**：测试是安全网  
4. **持续进行**：重构是常态，不是一次性任务  
5. **协作确认**：每个阶段都需要用户同意  

## 工作流概览

```
阶段 1：调研与分析
    ↓
阶段 2：测试覆盖评估
    ↓
阶段 3：代码异味识别
    ↓
阶段 4：制定重构计划
    ↓
阶段 5：增量实施
    ↓
阶段 6：回顾与迭代
```

---

## 阶段 1：调研与分析

### 目标
- 理解代码库结构与用途  
- 明确重构范围  
- 收集业务需求相关上下文  

### 向用户澄清的问题
开始前请先弄清：

1. **范围**：哪些文件/模块/函数需要重构？  
2. **目标**：要解决什么问题？（可读性、性能、可维护性）  
3. **约束**：有哪些区域**不应**改动？  
4. **时间压力**：是否会阻塞其他工作？  
5. **测试现状**：是否有测试？是否通过？  

### 行动项
- [ ] 阅读并理解目标代码  
- [ ] 识别依赖与集成点  
- [ ] 记录当前架构  
- [ ] 标注已有技术债标记（TODO、FIXME 等）  

### 产出
向用户呈现调研结论：
- 代码结构摘要  
- 已识别的问题区域  
- 初步建议  
- **请求确认是否继续**  

---

## 阶段 2：测试覆盖评估

### 为何需要测试
> 「没有测试就重构，就像不系安全带开车。」—— Martin Fowler

测试是安全重构的**关键前提**。没有测试，很容易引入缺陷。

### 评估步骤

1. **查找已有测试**
   ```bash
   # 查找测试文件
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **运行已有测试**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **查看覆盖率（若项目支持）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 决策点：询问用户

**若测试存在且通过：**
- 进入阶段 3  

**若测试缺失或不完整：**
向用户说明可选方案：
1. 先补测试（推荐）  
2. 在重构过程中增量补测试  
3. 无测试继续（有风险——需用户明确知情并承担）  

**若测试失败：**
- **停止**。先修复失败测试再谈重构  
- 询问用户：是否先修测试？  

### 测试编写指引（如需要）

对每个待重构的函数，确保测试覆盖：
- 主路径（正常流程）  
- 边界情况（空输入、null、边界值）  
- 错误场景（非法输入、异常）  

采用「红-绿-重构」循环：
1. 先写失败测试（红）  
2. 再让测试通过（绿）  
3. 最后重构

## 阶段 3：代码异味识别

### 什么是代码异味？
代码异味是更深层次问题的表象。它们不是 bug，但说明代码有改进空间。

### 常见代码异味检查清单

完整列表见 [references/code-smells.md](references/code-smells.md)。

#### 速查表

| 异味 | 表现 | 影响 |
|-------|-------|--------|
| **过长方法** | 方法超过约 30–50 行 | 难理解、难测试、难维护 |
| **重复代码** | 相同逻辑出现在多处 | 修 bug 要改多处 |
| **过大类** | 类承担过多职责 | 违背单一职责 |
| **依恋情结** | 方法更多使用其他类的数据 | 封装不佳 |
| **基本类型偏执** | 过度使用原始类型而非对象 | 缺少领域概念 |
| **过长参数列** | 方法参数过多（例如 4 个以上） | 难正确调用 |
| **数据泥团** | 多项数据总是一起出现 | 缺少抽象 |
| **switch 语句** | 复杂的 switch/if-else 链 | 难扩展 |
| **夸夸其谈未来性** | 「以防万一」的代码 | 不必要复杂度 |
| **死代码** | 未使用代码 | 困惑与维护负担 |

### 分析步骤

1. **自动化分析**（若有脚本）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **人工走查**
   - 系统通读代码  
   - 记录每种异味的位置与严重程度  
   - 按影响分类（严重/高/中/低）  

3. **排优先级**
   优先处理：
   - 阻碍当前开发的异味  
   - 导致缺陷或困惑的异味  
   - 位于高频变更路径上的异味  

### 产出：异味报告

向用户呈现：
- 已识别异味及位置列表  
- 每项的严重程度评估  
- 建议的处理顺序  
- **请用户确认优先级**

## 阶段 4：制定重构计划

### 选择重构手法

针对每种异味，从目录中选取合适重构。

完整列表见 [references/refactoring-catalog.md](references/refactoring-catalog.md)。

#### 异味与重构手法对应（示例）

| 代码异味 | 建议重构手法 |
|------------|---------------------------|
| 过长方法 | Extract Method, Replace Temp with Query |
| 重复代码 | Extract Method, Pull Up Method, Form Template Method |
| 过大类 | Extract Class, Extract Subclass |
| 依恋情结 | Move Method, Move Field |
| 基本类型偏执 | Replace Primitive with Object, Replace Type Code with Class |
| 过长参数列 | Introduce Parameter Object, Preserve Whole Object |
| 数据泥团 | Extract Class, Introduce Parameter Object |
| switch 语句 | Replace Conditional with Polymorphism |
| 夸夸其谈未来性 | Collapse Hierarchy, Inline Class, Remove Dead Code |
| 死代码 | Remove Dead Code |

### 计划结构

使用 [templates/refactoring-plan.md](templates/refactoring-plan.md) 中的模板。

对每一项重构写明：
1. **目标**：将改动哪些代码  
2. **异味**：要解决什么问题  
3. **重构**：采用哪种手法  
4. **步骤**：细化的微步骤  
5. **风险**：可能出什么问题  
6. **回滚**：如何撤销  

### 分阶段推进

**关键**：重构要分阶段、渐进引入。

**阶段 A：快赢**（低风险、高收益）
- 重命名变量以提升清晰度  
- 抽取显而易见的重复代码  
- 删除死代码  

**阶段 B：结构改进**（中等风险）
- 从长函数中抽取方法  
- 引入参数对象  
- 将方法移到合适类中  

**阶段 C：架构级变更**（风险更高）
- 用多态替代条件分支  
- 抽取类  
- 引入设计模式  

### 决策点：向用户展示计划

实施前：
- 展示完整重构计划  
- 说明各阶段内容及风险  
- 取得用户对各阶段的明确同意  
- **询问**：「是否从阶段 A 开始执行？」

## 阶段 5：增量实施

### 黄金法则
> 「改动 → 测试 → 绿了？→ 提交 → 下一步」

### 实施节奏

对每一步重构：

1. **前置检查**
   - 测试通过（绿）  
   - 代码能编译  

2. **只做一处小改动**
   - 按目录中的「做法」执行  
   - 保持改动最小化  

3. **验证**
   - 立即跑测试  
   - 检查编译错误  

4. **若测试通过（绿）**
   - 用清晰信息提交  
   - 进入下一步  

5. **若测试失败（红）**
   - **立即停止**  
   - 撤销该改动  
   - 分析原因  
   - 不清楚时询问用户  

### 提交策略

每次提交应：
- **原子化**：一个逻辑变更  
- **可回滚**：易于撤销  
- **说明清楚**：提交信息可读  

提交信息示例：
```
重构: 从 processOrder() 中抽取 calculateTotal()
重构: 将 'x' 重命名为 'customerCount' 以提升可读性
重构: 删除未使用的 validateOldFormat() 方法
```

### 进度同步

每个子阶段结束后向用户汇报：
- 已做改动  
- 测试是否仍通过  
- 遇到的问题  
- **询问**：「是否继续下一批？」

## 阶段 6：回顾与迭代

### 重构后检查清单

- [ ] 全部测试通过  
- [ ] 无新增告警/错误  
- [ ] 编译成功  
- [ ] 行为未变（必要时人工验证）  
- [ ] 需要时已更新文档  
- [ ] 提交历史清晰  

### 指标对比

在前后运行复杂度分析：
```bash
python scripts/analyze-complexity.py <file>
```

展示改进情况：
- 代码行数变化  
- 圈复杂度变化  
- 可维护性指数等变化  

### 用户验收

向用户呈现最终结果：
- 变更总览  
- 前后代码对比  
- 指标改善  
- 仍存技术债  
- **询问**：「你对这些改动是否满意？」  

### 后续

与用户讨论：
- 是否继续处理其他异味？  
- 是否安排后续重构？  
- 是否在其他位置做类似改动？

## 重要指引

### 何时暂停并询问用户

在以下情况应暂停并与用户确认：
- 对业务逻辑不确定  
- 改动可能影响对外 API  
- 测试覆盖不足  
- 需要做重大架构决策  
- 风险等级上升  
- 遇到未预期的复杂度  

### 安全规则

1. **无测试不重构**（除非用户明确接受风险）  
2. **不要大改**——拆成极小步骤  
3. **每次改动后必跑测试**  
4. **测试失败不继续**——先修复或先回滚  
5. **不要臆测**——有疑问就问  

### 不要做

- 不要把重构与加新功能混在同一批改动里  
- 不要在生产紧急事件期间重构  
- 不要重构自己尚不理解的代码  
- 不要过度设计——保持简单  
- 不要试图一次重构所有东西

## 快速入门示例

### 场景：过长方法且存在重复

**修改前：**
```javascript
function processOrder(order) {
  // 约 150 行代码，包含：
  // - 重复的校验逻辑
  // - 内联计算
  // - 职责混杂
}
```

**重构步骤：**

1. **确保** 对 processOrder() **已有测试**  
2. **抽取** 校验逻辑为 validateOrder()  
3. **跑测试** —— 应仍通过  
4. **抽取** 计算为 calculateOrderTotal()  
5. **跑测试** —— 应仍通过  
6. **抽取** 通知为 notifyCustomer()  
7. **跑测试** —— 应仍通过  
8. **回顾** —— processOrder() 现仅编排三个清晰函数  

**修改后：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
