## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（若非空）。用户可以指定：
- `create` — 从零新建 CLAUDE.md
- `update` — 改进现有 CLAUDE.md
- `audit` — 分析并汇报当前 CLAUDE.md 的质量
- 要创建/更新的具体路径（例如用于目录级说明的 `src/api/CLAUDE.md`）

## 核心原则

**大语言模型是无状态的**：CLAUDE.md 是唯一会在每次对话中自动纳入上下文的文件。它是 AI 智能体进入你代码库时的主要入门文档。

### 黄金法则

1. **少即是多**：前沿 LLM 大约能遵循 ~150–200 条指令。Claude Code 的系统提示词已占用约 50 条。请让 CLAUDE.md 聚焦且简短。

2. **普遍适用**：只收录与**每一次**会话都相关的内容。任务专属说明应放在其他文件中。

3. **不要把 Claude 当 Linter 用**：风格规范会膨胀上下文并削弱指令遵循能力。请改用确定性工具（prettier、eslint 等）。

4. **不要自动生成**：CLAUDE.md 是 AI 辅助体系中杠杆最高的一环。请手工撰写并审慎斟酌。

## 执行流程

### 1. 项目分析

首先分析当前项目状态：

1. 查找是否已有 CLAUDE.md：
   - 根目录：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 按目录：`**/CLAUDE.md`
   - 全局用户配置：`~/.claude/CLAUDE.md`

2. 识别项目结构：
   - 技术栈（语言、框架）
   - 项目类型（monorepo、单应用、库）
   - 开发工具（包管理器、构建系统、测试运行器）

3. 查阅现有文档：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 内容策略（WHAT、WHY、HOW）

围绕三个维度组织 CLAUDE.md：

#### WHAT — 技术与结构
- 技术栈概览
- 项目组织方式（对 monorepo 尤其重要）
- 关键目录及其用途

#### WHY — 目的与背景
- 项目是做什么的
- 为何做出某些架构决策
- 各主要组件的职责

#### HOW — 流程与约定
- 开发工作流（bun vs node、pip vs uv 等）
- 测试流程与命令
- 校验与构建方式
- 关键的「坑」或非显而易见的要求

### 3. 渐进式披露策略

对较大项目，建议创建 `agent_docs/` 目录：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中引用这些文件，例如：
```markdown
详细构建说明见 `agent_docs/building_the_project.md`
```

**重要**：用 `file:line` 引用代替代码片段，避免上下文过时。

### 4. 质量约束

创建或更新 CLAUDE.md 时：

1. **目标长度**：300 行以内（理想情况下 100 行以内）
2. **不要写风格规则**：去掉任何 lint/格式化说明
3. **不要写任务专属说明**：移到单独文件
4. **不要贴代码片段**：改用文件引用
5. **不要重复信息**：不要重复 package.json 或 README 里已有内容

### 5. 必备章节

结构良好的 CLAUDE.md 可包含：

```markdown
# 项目名称

一句话简介。

## 技术栈
- 主语言及版本
- 主要框架/库
- 数据库/存储（如有）

## 项目结构
[仅 monorepo 或复杂结构时需要]
- `apps/` — 应用入口
- `packages/` — 共享库

## 开发命令
- 安装：`command`
- 测试：`command`
- 构建：`command`

## 关键约定
[仅非显而易见且影响大的约定]
- 约定 1 及简要说明
- 约定 2 及简要说明

## 已知问题 / 易踩坑点
[反复让开发者碰壁的事项]
- 问题 1
- 问题 2
```

### 6. 应避免的反面模式

**不要包含：**
- 代码风格指南（请用 linter）
- 如何使用 Claude 的说明
- 对显而易见模式的冗长解释
- 复制粘贴的代码示例
- 泛泛的「最佳实践」（如「写干净代码」）
- 针对具体任务的指令
- 自动生成的内容
- 冗长的 TODO 列表

### 7. 校验清单

定稿前请确认：

- [ ] 300 行以内（最好 100 行以内）
- [ ] 每一行都适用于**所有**会话
- [ ] 无风格/格式化规则
- [ ] 无代码片段（使用文件引用）
- [ ] 所列命令已验证可用
- [ ] 复杂项目已采用渐进式披露
- [ ] 关键坑点已记录
- [ ] 与 README.md 无重复

## 输出格式

### 针对 `create` 或默认：

1. 分析项目
2. 按上文结构起草 CLAUDE.md
3. 提交草稿供审阅
4. 在确认后写入合适位置

### 针对 `update`：

1. 读取现有 CLAUDE.md
2. 对照最佳实践审计
3. 识别：
   - 要删除的内容（风格规则、代码片段、任务专属）
   - 要压缩的内容
   - 缺失的关键信息
4. 提交变更供审阅
5. 在确认后应用变更

### 针对 `audit`：

1. 读取现有 CLAUDE.md
2. 生成报告，包含：
   - 当前行数 vs 目标
   - 普遍适用内容占比
   - 发现的反面模式列表
   - 改进建议
3. **不要**修改文件，仅报告

## AGENTS.md 的处理

若用户要求创建/更新 AGENTS.md：

AGENTS.md 用于定义专用智能体行为。与 CLAUDE.md（面向项目上下文）不同，AGENTS.md 定义：
- 自定义智能体角色与能力
- 智能体专属指令与约束
- 多智能体场景下的工作流定义

应用类似原则：
- 保持聚焦、简短
- 使用渐进式披露
- 引用外部文档，而非嵌入大段内容

## 备注

- 写入前务必验证命令可用
- 不确定时宁可不写——少即是多
- 系统提醒会告诉 Claude，CLAUDE.md「可能相关也可能不相关」——噪音越多，越容易被忽略
- monorepo 最受益于清晰的 WHAT/WHY/HOW 结构
- 目录级 CLAUDE.md 应更加聚焦
